Operational Data Store (ODS)

Definition: An Operational Data Store (ODS) is a database designed to integrate data from multiple sources for additional operations on the data. It is a central database that provides a current, integrated view of data across the organization.

Key Characteristics of ODS:

  • Subject-Oriented: Organized around key business subjects
  • Integrated: Combines data from various operational systems
  • Current-Valued: Contains near real-time or current data
  • Volatile: Data is constantly updated as operational systems change
  • Detailed: Contains detailed, not summarized, data

Purpose of ODS:

  1. Provide a unified view of operational data
  2. Support operational reporting and decision-making
  3. Serve as a staging area for data warehouses
  4. Enable data consistency across systems
  5. Support real-time or near real-time business processes
Example: A retail company might have an ODS that integrates data from point-of-sale systems, inventory management, and customer relationship management (CRM) to provide a complete view of customer transactions.
Important: ODS is different from a data warehouse. While a data warehouse is optimized for analysis and historical data, an ODS is optimized for operational reporting and current data.

Data Mart

Definition: A Data Mart is a subset of a data warehouse that is focused on a specific business line or department. It contains a curated selection of data tailored to the needs of a particular group of users.

Types of Data Marts:

  • Dependent Data Mart: Built from an existing data warehouse
  • Independent Data Mart: Built directly from source systems without a data warehouse
  • Hybrid Data Mart: Combines data from a data warehouse and other sources

Benefits of Data Marts:

  1. Faster implementation compared to full data warehouses
  2. Tailored to specific departmental needs
  3. Improved performance for specific queries
  4. Easier for business users to navigate
  5. Lower cost than enterprise data warehouses
Example: A sales data mart would contain only sales-related data (transactions, customers, products) specifically designed for the sales department's reporting and analysis needs.

Data Mart vs Data Warehouse:

  • Scope: Data Mart (Departmental) vs Data Warehouse (Enterprise-wide)
  • Data: Data Mart (Subject-specific) vs Data Warehouse (Comprehensive)
  • Size: Data Mart (Smaller, GBs) vs Data Warehouse (Larger, TBs/PBs)
  • Implementation Time: Data Mart (Weeks/Months) vs Data Warehouse (Months/Years)
Tip: When designing a data mart, focus on the specific needs of the target users. Include only the data they need in a structure that matches their business processes.

Data Cleansing

Definition: Data Cleansing (or Data Cleaning) is the process of detecting and correcting (or removing) corrupt, inaccurate, or irrelevant parts of data. It improves data quality and reliability.

Common Data Quality Issues:

  • Missing Values: Empty or null fields
  • Inconsistent Formatting: Different formats for the same data (e.g., dates, phone numbers)
  • Duplicate Records: Multiple entries for the same entity
  • Spelling Errors: Typos and misspellings
  • Outliers: Values that are significantly different from others
  • Incorrect Data Types: Numbers stored as text, dates stored as strings

Data Cleansing Process:

  1. Audit: Analyze data to identify quality issues
  2. Plan: Define rules and methods for cleansing
  3. Execute: Apply cleansing rules to the data
  4. Verify: Check if cleansing improved data quality
  5. Document: Record the cleansing process for future reference

Data Cleansing Techniques:

  • Parsing: Breaking data into components
  • Standardization: Converting data to consistent formats
  • Matching: Identifying related or duplicate records
  • Enrichment: Adding missing information from external sources
  • Validation: Checking data against predefined rules
Example - Before Cleansing:
Customer records: "John Smith", "john smith", "J. Smith", "Smith, John"

After Cleansing:
All standardized to: "Smith, John"
Important: Data cleansing is an ongoing process, not a one-time activity. As new data enters the system, it should go through validation and cleansing routines.

BigQuery Introduction

Definition: Google BigQuery is a fully-managed, serverless data warehouse that enables scalable analysis over petabytes of data. It is a Platform-as-a-Service (PaaS) that supports queries using ANSI SQL.

Key Features of BigQuery:

  • Serverless: No infrastructure to manage
  • Highly Scalable: Can handle petabytes of data
  • Fast: Uses Google's infrastructure for rapid query execution
  • Cost-Effective: Pay only for the data processed and stored
  • Secure: Built-in security and compliance features
  • Integrated: Works with other Google Cloud services

BigQuery Architecture:

  1. Storage Layer: Uses Google's Colossus distributed file system
  2. Compute Layer: Uses Dremel query engine for fast execution
  3. Borg: Google's cluster management system allocates resources
  4. Jupiter: Google's network fabric connects all components

BigQuery Pricing Model:

  • On-Demand: Pay per query based on bytes processed
  • Flat-Rate: Monthly fee for predictable workloads
  • Storage Pricing: Cost for active and long-term storage
  • Streaming Inserts: Additional cost for real-time data ingestion
Example Use Cases:
  • Analyzing years of sales data to identify trends
  • Processing web analytics data for marketing insights
  • Running machine learning models on large datasets
  • Creating real-time dashboards from streaming data
Tip: BigQuery automatically scales to handle your data volume and query complexity. You don't need to worry about provisioning resources or managing clusters.

Creating Datasets in BigQuery

Definition: In BigQuery, a dataset is a top-level container that is used to organize and control access to your tables and views. A dataset is contained within a specific Google Cloud project.

Dataset Components:

  • Tables: Contain the actual data
  • Views: Virtual tables defined by a SQL query
  • Models: Machine learning models created in BigQuery ML
  • Routines: User-defined functions (UDFs) and stored procedures

Creating a Dataset via BigQuery UI:

  1. Open the BigQuery console in Google Cloud
  2. Select your project in the Explorer panel
  3. Click the three dots next to your project name
  4. Select "Create dataset"
  5. Provide a Dataset ID (name)
  6. Select a Data location (region)
  7. Set default table expiration (optional)
  8. Choose encryption options
  9. Click "Create dataset"

Creating a Dataset via SQL:

CREATE SCHEMA IF NOT EXISTS `my-project.my_dataset`
OPTIONS(
  location = "us",
  default_table_expiration_days = 30
);

Creating a Dataset via bq Command-Line Tool:

bq mk --dataset --location=US --default_table_expiration=3600 my-project:my_dataset

Dataset Best Practices:

  • Use descriptive names for datasets
  • Group related tables in the same dataset
  • Set appropriate table expiration policies
  • Choose the location closest to your users
  • Use consistent naming conventions across projects
Important: The dataset location determines where the data is stored and cannot be changed after creation. Choose carefully based on compliance requirements and user locations.
